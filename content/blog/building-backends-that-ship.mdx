Most backend advice is about scale you'll never hit. This is about the boring
stuff that actually breaks in production — and how to get ahead of it.

## Model the data first

Before a single endpoint, I sketch the tables. Good schemas make good APIs
almost automatic; bad ones leak complexity into every layer above.

- Index the columns you filter and join on — measure with `EXPLAIN ANALYZE`.
- Prefer explicit foreign keys; let the database enforce what it can.
- Keep write paths narrow so invariants live in one place.

## Make failure boring

Every external call fails eventually. Wrap them in timeouts, retries with
backoff, and idempotency keys so a retry never double-charges anyone.

> The goal isn't zero failures — it's failures that don't page you at 3am.

## Ship small, ship often

A feature behind a flag in production teaches you more in a day than a week of
staging. Instrument it, watch the p95, and iterate.

That's it. No microservices, no Kafka — until you actually need them.
